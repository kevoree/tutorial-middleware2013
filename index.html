Means of reflection in Distributed Environment

Direct access to reflexive layer is an issue in several distributed architectures
  • Sporadic nodes : direct reflexive access is not alwaysreachable pvgup
  • Heterogeneous nodes : direct access is not always provided by restrictive runtime pvtuq
  • Embedded nodes : direct access can be an issue for the battery life p vu

Heterogeneous dissemination strategy

Distributed systems usually organized in clusters (group of nodes) Considering a cluster of 3 nodes p sharing a configuration model :
  • Sporadic context : N1 can update N2 without synchronizing N3
    N1 N2 N3
  • Cluster context : N1 must synchronize N2 & N3 in the same logical time
    N2 N1 N3
  • Opportunistic nodes : Depending on previous configurations, A can synchronize B or C
    N2 N1 N3

Architectural abstraction needed !

Component close isolation

Component : Closed entity, part of a system, has responsibility

  • Close isolation = CA cannot directly call any method of CB
  • Components can depend on third-parties libraries
  • Ports are the borders of components
  • Communications are supported by Channels (C1)

Port Type

  • MessagePortType
    • Input / Output
    • Asynchronous InOnly/OutOnly
    • Event driven
  • ServicePortType
    • Provided/Required
    • Synchronous InOut MEP
    • RMI-like

Channel Type

  • Extract communication semantics from component
  • Explicit complex distribution scenario
  • Model communication semantics

Channel Type

  • ChannelType defines
    • Lower and Upper bindings bounds
    • Lower and Upper distribution bounds
  • Can support
    • Messages only or Services only
    • Both
  • Implement the distribution/call mechanism
  • Can implement complex behaviour
    • Priority, parallel, sequence, first answer, collection, etc...

@ChannelType
public class SyncBroadcast implements ChannelDispatch {

    @KevoreeInject
    ChannelContext channelContext;

    @Override
    public void dispatch(Object payload, Callback callback) {
        for (Port p : channelContext.getLocalPorts()) {
            p.call(payload, callback);
        }
    }
}

    • Smartphones (android, Java)
    • Low consumption Cloud VM based on ARM

    Challenges of adaptation on μC

    μC nodes not powered enough to support all adaptation strategies!
    Dynamic adaptation on μC faces several challenges
    • For comfort and critical uses, downtime must be minimal
    • Volatile memory usage must fit in few kB
    • Persistent memory writes are limited
    • Reconfiguration must be statefull, to recover from power failure
    use more powerful sibling to pre-compile adaptations

    μKev Component model

    • Type definition/instance (what/where)
    • Component with ports (in/out) and parameters
    • Communication channels between ports (bindings)
    • Write code (C/Processing/Java) Instantiate in a model
